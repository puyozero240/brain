# 5年わからなかったGitが、GitHubを捨てたら一瞬で理解できた話
![](/images/git_github_zukai_note.png)

「GitHubを使えるようにならないといけない」  
「GitHubは絶対使ったほうがいい」

そう見聞きしてきてどれくらいたっただろう。多分5〜6年、いやもっとかもしれない。それだけの長い期間私はGitHubの概念が腑に落ちず、もやもやしていた。

しかし今日、「あぁ、そういうことだったのか」とカタルシスがおき、やっと腑に落ちる形で理解が進んだ。私自身がコードをガッツリ書くエンジニアじゃないからこそ、ここまで理解に苦しんだのだと思う。

今回の記事では、5年以上もの間「なんとなくわかるけど、わからんなぁ」と理解に苦しみ続けた私が、「ようやくGitHubを理解できたぞ！」という喜びに身を任せ筆を走らせようと思う。

きっかけは『Markdown x GitHub』という掛け合わせだった。

## GitHub 一つの言葉で 2概念(五七五)

まずは理解するにあたり、GitHubというものがなぜこうもわかりにくいのかということを話しておきたい。腑に落ちた今だからこそ言語化ができる。

私がGitHubの理解に苦しんだ理由を整理すると、以下の2つが主な原因だと考えられる。

- GitHubは『Git(バージョン管理) +  GitHub(サービス名)』で出来ている
- GitHubの語呂のよさが皮肉にも理解を妨げる
- 『プログラミングのコーディング』と『Git/GitHubへの理解』という3重の負荷がエグい


### GitHubは『Git(バージョン管理) +  GitHub(サービス名)』で出来ている

根本的な問題はひとことに"GitHub"といってしまっていることだと私は思う。

『GitHub』を学習していると『Git』と『GitHub』という言葉が入り乱れている(ように見える)。ここが問題なのだ。

もちろん各教材でちゃんと説明はされてるのだろうが、初学者は言葉が頭に残らない。その結果、我々非エンジニア(初学者)は『GitHub』という語呂の良い、しかし本質的ではない音韻(音のひびき)が頭に残る。

これが、迷宮の入り口となる。


### GitHubの語呂のよさが皮肉にも理解を妨げる

この『GitHub』という名前は非常に覚えやすいが、これが理解を遠ざけてしまう一つの原因なのだと思う。

後述するが『Git』を理解すればこの概念は理解が進む。しかし、『GitHub』という言葉が頭に残っている我々は、この『GitHub』というキーワードで検索をしてしまう。

その結果、本来理解するべきはずの『Git』の概念を調べることができずわからず仕舞いという悲しい結果になってしまうわけだ。

理解が進んだ今だからこそわかるが、各記事を見てみるとちゃんと『Git』と『GitHub』の概念を区別して説明している。

[『GitHub とは 初心者』の検索結果](https://www.google.com/search?q=github+%E3%81%A8%E3%81%AF+%E5%88%9D%E5%BF%83%E8%80%85)

しかし理解できていなかった時の私は『GitHub』という単語が頭に残っているためこの『Git』の解説を読み飛ばしていた。読んでいたとしても「GitHubってGitともいうんだなぁ」と自分でごちゃ混ぜにして読んでしまう。これではわかる訳がない。

理解した今なら言える。まず覚えるべきは間違いなく『Git』の方だ。

そして『Git/GitHub』の解説を見ると、ほぼ100%の確率でプログラミングの文脈で解説されている。これまた理解を遠ざけてしまうのである。


### 『プログラミングのコーディング』と『Git/GitHubへの理解』という3重の負荷がエグい

先述したとおり『Git』を学ぼうとするとほぼ必ず、100%の確率でプログラミングの文脈のなかで解説をされることになる。

こういった解説は3つの要素の理解を同時に求められる。

* コードの理解
* Gitの理解
* GitHubの理解

この説明のされ方をすると初学者にはかなり負担だ。そして次第にどっちの話をしてるのかわからなくなってしまう。そしてついには「あ〜なんとなく…わかった気がする（わかってない）」という状態に落ち着く。何一つわかっていないというのに。


## 『Markdown x Git/GitHub』という組み合わせでGitの理解に集中できた。

つまり私は、AIの回答によって偶然にも『Markdown x Git/GitHub』という組み合わせに出会った。この組み合わせによって先述した問題から解き放たれ、本質であるGitの理解に集中できたのだ。

今はこういう形で理解している。

* バージョン管理はGitだけでできる
* GitHubはただ共有する場所

```python
# バージョン管理はGitだけでできる
Git
├── init (初期設定)
├── add (記録準備)
└── commit (記録)

# GitHubはただ共有する場所
GitHub
├── push(アップロード)
└── pull(ダウンロード)
```


## バージョン管理はGitだけでできる

これまで散々書いてきているのでもうおわかりだとは思うが、私は今までGitとGitHubをごちゃ混ぜにしていた。『バージョン管理』はGitだけで、しかもローカルでだけできるのだ。

つまり、巷の解説でよく解説で言われている

```bash
# gitの初期設定
git init

# 編集をステージング(記録する準備を)する
git add .

# バージョンをコミット(記録)する
git commit -m "commit message"
```

というバージョン管理は、『GitHub』は不要なのである。

ローカルだけで、Gitだけで、『バージョン管理』そのものはできてしまうのである。『GitHub』は一旦脇に置いておこう。


## Gitがわかれば6割くらいわかる

Gitの概念さえ理解してしまえば、実はGitHubを使うための準備の6割は終わっていると言ってもいいとおもっている。なぜなら日々の作業である「ファイルを作成・変更し記録する」という行為自体は、すべてローカル（自分のPCの中）だけで完結してしまうからだ。

「プッシュする」だの、「プルリクを送る」だの、そんなことは個人のメモ管理においては一度忘れてしまっていい。

ただひたすらに `git add` し `git commit` する。これだけで「いつ」「誰が」「何をしたか」という履歴は完璧に残すことができるのだから。本来の目的の『バージョン管理』はこれだけでできるのだから。


### ブランチとマージについては覚えとくと便利

より便利にバージョン管理をするのであれば、ブランチとマージも覚えておくと便利だ。


#### ブランチ

元のバージョンに影響を与えずいろいろ試すことができる。平行世界で編集ができるイメージだろうか。平行世界では何をしても元いた世界に何の影響もない。やりたい放題だ。

```bash
# ブランチ作成と切り替えを同時にやる
# <新しいブランチ名> は任意
git switch -c <新しいブランチ名>

# 作成したブランチを確認する
git branch
```

"平行世界"というブランチを作るとすると

```bash
git switch -c 平行世界
```

とすればよい。


#### マージ

平行世界(ブランチ)で攻めた作業を進め、そこでの方針が「ええやん、これでいこうや」と思えたときは、マージする。

そうすると平行世界での編集を元のバージョン(元の世界)に反映することができるわけだ。

```bash
git merge <ブランチ名>
```

ややこしくなってくるので、ここからの説明は表現を下記のように言い換えて説明する。

  * 元のバージョン = **元の世界**
  * ブランチ = **平行世界**

マージする際は平行世界から元の世界に戻ってから行う必要がある。

もし仮に、平行世界にいる状態でマージをしてしまうと、意図しない取り込まれ方をしてしまう。

なので実際にマージするときは**必ず**元の世界に戻ってからマージを行う必要がある。

```bash
# 元のバージョン(元の世界)に戻る
git switch 元の世界

# 平行世界を元の世界にマージする
git merge 平行世界
```

なんとなく…イメージできただろうか。私はこれで理解しているが、伝わるかは怪しい。余計に説明が分かりにくくなっていたら申し訳ない。

## 残りの4割はGitHubなどのサービスへの理解

残りの4割であるGitHubは単純に『バックアップ』だと思えばいい。
ローカルで積み上げた `git commit` の履歴を、インターネット上の倉庫（GitHub）に転送する。それが『プッシュ』だ。

『プッシュ』することで、ローカルの歴史が『GitHub』に同期される。

チーム開発であればここから『プル』だのなんだのといった複雑な話が出てくるが、いったん触れなくてもいいと思う。GitHubは「高機能なバックアップ先」くらいに捉えておけばいいだろう。


## Gitでテキストのローカル管理からはじめよう

「『Git/GitHub』を理解できた！」という喜びに身を任せ筆をはしらせた。多少文章が散らかっていて申し訳ないと思うが、これは書かずにはいられなかった。

理解してみればとてもシンプルな話だった。

  * \*\*Git (ローカル)\*\*は作業履歴を残す場所（主戦場）
  * \*\*GitHub (リモート)\*\*はそれを保存しておく場所（倉庫）

私はこれをごちゃ混ぜにしていたから理解できなかった訳だ。

まずはローカルで `git add` と `git commit` を使って、バージョン管理を息をするようにできるようになること。

これができれば、GitHubはただの「保存先」として自然と使えるようになっているはずだと考えている。
